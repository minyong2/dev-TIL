# CI/CD (지속적 통합 및 지속적 배포) 개념 정리

## 🚀 CI (Continuous Integration)
- **정의:** 개발자의 코드 변경 사항을 정기적으로 자동 빌드 및 테스트하여 공유 저장소에 통합하는 것.
- **주요 단계:** 코드 커밋 -> 자동 빌드 -> 자동 테스트.
- **핵심 가치:** 코드 충돌 방지 및 초기 버그 발견.

## 🚚 CD (Continuous Delivery & Deployment)
- **Continuous Delivery:** 빌드된 결과물을 검증 후 배포 대기 상태로 유지 (수동 배포 승인).
- **Continuous Deployment:** 검증된 결과물을 운영 서버에 자동으로 직접 반영 (완전 자동 배포).
- **핵심 가치:** 빠른 서비스 출시 및 배포 자동화.

## 🛠 대표적인 CI/CD 도구
- **GitHub Actions:** GitHub에서 제공하는 워크플로우 자동화 도구.
- **Jenkins:** 가장 널리 쓰이는 오픈소스 자동화 서버.
- **GitLab CI/CD:** GitLab 내장형 파이프라인 도구.

---
## 🏗️ Jenkins (젠킨스)
가장 대중적인 오픈소스 CI/CD 도구로, 다양한 플러그인을 통해 거의 모든 환경의 자동화를 지원합니다.

### 🔄 작동 방식
1. **Trigger:** 개발자가 코드를 Push (Webhook 발생).
2. **Checkout:** 젠킨스가 저장소에서 소스 코드를 가져옴.
3. **Build & Test:** 미리 정의된 스크립트(Pipeline)에 따라 빌드 및 테스트 수행.
4. **Result:** 성공 시 배포 진행 및 메신저(Slack 등)로 결과 알림.

### ✅ 장점
- **무료:** 오픈소스이므로 비용 부담이 적음.
- **확장성:** 1,000개가 넘는 방대한 플러그인 생태계.
- **커스터마이징:** 복잡한 배포 시나리오도 직접 스크립트로 짤 수 있음.

### ⚠️ 단점
- **직접 관리:** 서버를 직접 구축하고 유지보수해야 함.
- **초기 설정:** GitHub Actions 같은 클라우드 서비스에 비해 설정이 복잡함.

---
## 🛠️ Jenkins 구축 단계 (Docker 기준)

### 1. Docker 설치 및 젠킨스 컨테이너 실행
- Docker를 통해 복잡한 환경 설정 없이 젠킨스 서버를 격리된 환경에서 구동.
- 데이터 영속성을 위해 `-v (Volume)` 옵션을 반드시 사용하여 설정 데이터를 보존.

### 2. 초기 잠금 해제 (Unlock)
- 최초 실행 시 생성되는 `initialAdminPassword`를 통해 관리자 권한 인증.

### 3. 플러그인 설정
- **Pipeline:** CI/CD 흐름을 정의하는 핵심 플러그인.
- **Git / GitHub:** 소스 저장소 연동을 위한 필수 도구.
- **Credentials:** API 키나 비밀번호를 안전하게 관리.

### 4. 파이프라인(Pipeline) 구성
- `Jenkinsfile`을 프로젝트 루트에 생성하여 빌드-테스트-배포 단계를 코드로 관리 (Pipeline as Code).
