### 개인정보 제공동의 ✔️
- 인바운드 기간 설정 , 기간종료 후 미수집자 분류 
    - [✔️] 인바운드 기간 설정 테이블 필요
    - 스케줄 주기는?

- 미수집자 존재 > 아웃바운드 전환 (테이블to테이블) 
    - [✔️] 아웃바운드 테이블 UPSERT (status 체크)
        - `ars_call_status_code` null = '발송대기'
    - [✔️] 인바운드 테이블 DELETE 

- 아웃바운드 미수집자 자동 발송
    - 발신 요청 파라미터 중 시나리오ID는 무엇?
    - [✔️] ARS API CALL
    - [✔️] 이력테이블 INSERT , 아웃바운드 테이블 UPSERT

- 최종 결과 수신 API 생성
    - [✔️] 컨트롤러 생성 ("/outboundResult.do")
        - ARS 모듈 측 결과 콜백 API 엔드포인트 수정
    - [] DOCKER 설치 및 ARS 모듈 response 테스트
        - ARS 모듈 측 List 수신 로직 수정 필요 (객체 하나만 받게 되어있음)
    - [] 수락 > 이력테이블 기록, 아웃바운드 테이블 DELETE (result:agree)
    - [] 거절 > 미등록 (수동전환) , 아웃바운드 테이블 남겨두기
        - 완료 됐는데 제공동의 N이면 수동

- 수동 발신
    - [✔️] 컨트롤러 생성 ("/retryOutboundCall.do")
    - [✔️] 서비스 생성 (retryOutboundCall)
    - ARS API CALL
        - List에 담아서 전송
    - [] 수동 발신 > STATUS 변경 시 아웃바운드 테이블 UPDATE , 이력 INSERT
        - 아웃바운드 테이블은 가장 최신 상태값을 갖고 있는 것

- 미등록 대상 > 미등록번호에서 메시지 청취 요청 시 개인정보 수집‧이용 동의 안내 절차가 진행되어야 함

### 테스트 프로세스 💻
1. ARS 자동발송 테스트는 아웃바운드 테이블에만 데이터가 있어도 가능.
    - [✔️] 아웃바운드 테이블 조회 > [✔️] ARS CALL > [✔️] 아웃바운드 테이블 업데이트 [] HISTORY INSERT 
        > CALL RESPONSE > HISTORY UPDATE(historyNo)

2. 인바운드 테스트는 데이터가 잘 이관되는지만 확인하면 됨.
